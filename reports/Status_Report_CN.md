## 执行摘要 (Executive Summary)

“舞蹈姿态评分与辅助训练系统”项目已顺利完成核心开发阶段（第一阶段至第三阶段）。项目的主要目标是开发一个实时的计算机视觉应用程序，能够分析人体舞蹈动作并提供量化反馈。截至目前，项目已交付了一个功能健壮的桌面应用程序，其特点包括双重加权评分算法（结合关节角度和 Procrustes 形状分析）、多线程视频处理架构以及现代化的以用户为中心的图形界面。虽然与最初的提案存在细微偏差——特别是关于最初基于 Web 的部署范围与当前侧重于桌面的焦点——但项目的主要里程碑仍按计划进行。该系统在姿态估计方面表现出高准确性，并通过实时评分图表和节奏分析提供可操作的反馈。剩余工作集中在高级智能功能上，例如用于节奏精度的动态时间规整 (DTW) 和基于大语言模型 (LLM) 的反馈生成，这些都计划在最后阶段进行。

## 项目描述 (Project Description)

本项目旨在利用人工智能普及专业舞蹈训练。系统使用标准网络摄像头捕捉用户的动作，并实时将其与专业教练的参考视频进行比较。通过采用 MediaPipe 框架进行姿态估计和自定义几何算法进行评分，应用程序提供关于姿态准确性和时机的即时、客观反馈。该解决方案解决了个人舞蹈指导的高成本和有限可用性问题，为自我提升提供了一种易于获取的工具。

## 项目目标、范围、方法、主要里程碑和交付成果概览

### 目标 (Objectives)
1.  **实时性能**：在标准消费级硬件上实现至少 30 FPS 的处理速度。
2.  **评分准确性**：开发一个能够以高精度区分正确和不正确姿势的评分模型，专门针对肢体角度和整体身体形状。
3.  **用户体验**：创建一个直观的界面，在不干扰用户的情况下提供即时的视觉和数字反馈。

### 范围 (Scope)
项目范围包括开发基于 Python 的桌面应用程序。主要功能包括视频文件和摄像头输入处理、姿态估计、同步播放、双标准评分引擎和课后表现报告生成器。像 AR 幽灵叠加和基于音频的节拍检测等高级功能被视为最后阶段的扩展目标。

### 方法 (Approach)
开发遵循迭代、敏捷的方法论。
*   **第一阶段**：核心基础设施搭建（MediaPipe 集成，基础 UI）。
*   **第二阶段**：算法优化（双重加权评分模型）。
*   **第三阶段**：UI/UX 增强（暗黑模式，实时图表）和性能优化（多线程）。
*   **第四阶段**：智能扩展（DTW，LLM 集成）。

## 对反馈/建议的回应 (Response to Feedback/Recommendations)

**提案反馈：** “最初的提案侧重于 Web 部署，这可能会引入对实时反馈至关重要的延迟问题。建议优先考虑本地处理性能。”

**回应：**
我们将主要的部署目标从基于 Web 的应用程序转向了使用 PyQt5 的原生桌面应用程序。这一转变允许直接访问硬件资源，显著减少了网络传输造成的延迟。虽然保留了后端服务模块用于潜在的未来混合模型，但核心开发工作（90%）致力于优化本地视频读取和姿态处理线程。这一决定已被证明是有效的，目前的测试显示稳定的 30 FPS 性能，端到端延迟小于 100ms。

**提案反馈：** “简单的欧几里得距离可能无法捕捉舞蹈动作的细微差别，例如关节角度。”

**回应：**
我们实施了**双重加权评分模型**（详见进度报告第 2.2.1 节）。该模型结合了加权 Procrustes 距离（针对全局形状）和余弦相似度（针对局部关节角度）。我们为肢体末端（手腕、脚踝）分配了更高的权重（1.5倍 - 2.0倍），以专门解决关于动作细微差别的反馈。

## 与原始提案的变更或偏差 (Variations or Deviations from Original Proposal)

1.  **从 Web 优先转向桌面优先架构**
    *   *原计划*：React/Vue 前端配合 Python 后端。
    *   *现况*：PyQt5 桌面应用程序。
    *   *理由*：通过 HTTP 实时同步两个视频流（用户和参考）引入了不可接受的抖动和延迟。桌面方法确保了帧级完美的同步，这对舞蹈评分至关重要。

2.  **推迟音频节拍检测**
    *   *原计划*：包含在第二阶段。
    *   *现况*：移至第四阶段（未来工作）。
    *   *理由*：优先考虑视觉节奏检测（基于帧的时序分析），以确保核心视觉评分系统的稳健性。音频处理需要复杂的信号分析库，如果不谨慎实施，可能会危及主线程的稳定性。

3.  **引入“幽灵叠加”概念**
    *   *原计划*：未提及。
    *   *现况*：添加到路线图。
    *   *理由*：用户测试表明，并排比较有时不足以进行自我纠正。AR 风格的叠加被确定为未来迭代的高价值功能。

*   **问题**：视频宽高比失真。非 16:9 视频最初被拉伸。
    *   *缓解措施*：我们在视频显示组件中实施了黑边填充策略，以在居中内容的同时保持宽高比。
*   **问题**：线程安全。最初版本在快速切换摄像头时会崩溃。
    *   *缓解措施*：我们在视频读取器中引入了严格的互斥锁，并为所有跨线程通信引入了信号槽机制。
